subsubsection ‚ÄπMonadic Formalization‚Ä∫

text ‚Äπ‚Äπ('ret,'ex,'RES_N,'RES) state‚Ä∫ represents any potential returning state of a program.

‚ñ™ ‚ÄπSuccess v r‚Ä∫ represents a successful returning state with return value ‚Äπv‚Ä∫ and resulted resource
  state ‚Äπr‚Ä∫.

‚ñ™ ‚ÄπAbnormality v r‚Ä∫ represents the computation throws an exception of value ‚Äπv‚Ä∫, at the
  moment when the state of the resources is ‚Äπr‚Ä∫.

‚ñ™ ‚ÄπNonTerm‚Ä∫ represents the execution does not terminate.

‚ñ™ ‚ÄπInvalid‚Ä∫ represents the computation is invalid.
  It defines the domain of valid programs, which are the ones that never never generates
  an execution path that results in ‚ÄπInvalid‚Ä∫.

‚ñ™ ‚ÄπAssumption_Violated‚Ä∫ represents the computation that results in an execution path that is not
  considered or modelled by the trust base.
  For example, the formalization of the allocation instruction may assume the size of the object
  to be allocated is always less than the size of the address space (e.g., ‚Äπ2‚áß3‚áß2‚Ä∫ bytes).
  In another case users may assume the size of their objects is representable by ‚Äπsize_t‚Ä∫.
  ‚ÄπAssumption_Violated‚Ä∫ enables an easy way for semantic assumptions, e.g., to assume ‚ÄπP‚Ä∫,
  \[ ‚Äπif P then do-something else return Assumption_Violated‚Ä∫ \]
  ‚ÄπAssumption_Violated‚Ä∫ is admitted by any post-condition, i.e.,
  \[ {‚ä§} return Assumption_Violated {‚ä•} \]
  Computations returning ‚ÄπAssumption_Violated‚Ä∫ are trusted by the trust base.
  This additional increment in the trust base is controllable, because whether and where to
  use the ‚ÄπAssumption_Violated‚Ä∫, this is determined by the formalization of instruction semantics,
  which belongs to the trust base already.
‚Ä∫

text ‚ÄπProcedure is the basic building block of a program on the semantics formalization.
It represents a segment of a program, which is defined inductively,

‚ñ™ Any instruction instantiated by any arguments is a procedure
‚ñ™ Any lambda abstraction of a procedure is a procedure
‚ñ™ Sequential composition ‚Äπf ‚§ú g‚Ä∫ is a procedure iff ‚Äπf, g‚Ä∫ are procedures.
‚ñ™ A control flow combinator combines several procedures into a procedure

Control flow combinator does not include the sequential composition combinator ‚Äπ‚§ú‚Ä∫.

Any function, routine, or sub-routine in high level languages is a procedure but a procedure
does not necessarily corresponds to them nor any basic block in assemble representations.

A procedure is not required to be a valid program necessarily.
Procedure is only a syntax structure and the semantics of invalid operations or programs
is expressed by returning ‚ÄπInvalid‚Ä∫.
